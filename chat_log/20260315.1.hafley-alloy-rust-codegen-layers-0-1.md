# Session: hafley-alloy Rust codegen layers 0-1 + symbols

## Goal
Build an Alloy-based Rust language package (hafley-alloy) for code generation, replacing dirty custom string-template codegen at work. TypeSpec models in, clean Rust out, with automatic `use` import generation via Alloy's binder/refkey system.

- 51 tests passing across 6 test files
- Depends on `@alloy-js/core@0.23.0-dev.12` (need dev version for `createScope`/`createSymbol`)

- Layer 0 (name policy) and Layer 1 (declaration components) complete as pure renderers

## Problem/Context
- Work has dirty string-template Rust codegen with no cross-file references, no import generation, no scope awareness
- Alloy framework solves this with binder/refkey/scope but has no Rust language package
- Go package at v0.2.0 in alloy monorepo served as reference implementation
- npm @alloy-js/core@0.22.0 tree-shakes out createScope/createSymbol -- must use 0.23.0-dev.12+

## Solution/Approach
Layered build, simplest-to-test first:
- Layer 0: Name policy (keyword r# escape only, NO case renaming)
- Layer 1: Declaration components (struct, enum, fn, impl) as pure JSX string renderers
- Layer 1.5: Symbol/scope class hierarchy + factories
- Layer 2: SourceFile + CrateDirectory (mod tree = file tree)
- Layer 3: Reference + UseDeclaration (binder resolves cross-file refs)

## Key Insights
- NEVER auto-rename identifiers for casing -- cross-repo grep > convention enforcement (saved to memory)
- Alloy List uses joiner/comma/hardline props not separator -- use For with joiner for inline lists
- AppendFile + AppendRegion in alloy core already does zone-based partial codegen with alloy-{name}-start/end sigils -- free feature
- Rust mod tree = file tree means binder pathDown should map 1:1 to use crate:: paths
- Go package pattern: Symbol subclass + Scope subclass + factories + contexts, all numbered by dependency order
- Alloy reactivity is @vue/reactivity, track()/trigger() for reactive symbol properties
